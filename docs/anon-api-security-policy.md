# 匿名 API 抑止セキュリティ方針

_Last updated: 2026-05-04_

kabu-ai.jp が匿名(未ログイン)ユーザーに有料リソース(LLM 等)を提供する API について、IP 偽装・並列攻撃・Cookie 偽装に対するセキュリティ方針を定める。具体的な実装ではなく、**何を守り・どこまでを許容し・どの層で何を達成するか**の枠組み。

---

## 1. 目的とスコープ

匿名で利用可能な API に対し、**「人間 1 人あたり 1 日 N 回」を経済合理性のラインで守る**。完全な abuse 撲滅は目指さない。LLM/外部 API 等のコストが攻撃者にペイしない閾値まで上げることが目標。

対象 API:
- `/api/agent-portfolio` ほか、未ログインで叩ける課金リソース API すべて

非対象:
- 認証済みユーザーの API（別の認可・課金スキームで管理）

---

## 2. 脅威モデル

| 攻撃者プロファイル | 想定手段 | 抑止優先度 |
|---|---|---|
| カジュアルな悪用者 | localStorage クリア、Cookie 削除 | **必須で防ぐ** |
| 個人スクリプトキディ | curl/Python スクリプト連打、ヘッダー偽装 | **必須で防ぐ** |
| 一般的な Bot | Bot framework、IP 単一・UA ローテーション | **必須で防ぐ** |
| 中程度の攻撃者 | VPN・プロキシ切替、IPv6 アドレス回転 | **コストで割に合わなくする** |
| 経済合理的な攻撃者 | 住宅プロキシ網 + 実ブラウザ automation + Turnstile solver | **受容**（経済閾値で抑制） |
| 国家級・標的型 | スコープ外 | スコープ外 |

---

## 3. 防御原則

### 3.1 多層防御（Defense in Depth）
単一層が破られても、別の層が止める設計にする。Cookie 偽装が成立しても DB クオータが守る、UA 偽装が成立しても IP レイヤで止まる、というように。

### 3.2 サーバー側で最終判定
クライアント側の状態（localStorage、JS 変数、クライアントから送られる任意ヘッダー）は**抑止のキーにしない**。すべての最終判定はサーバー側のサーバー権限ある状態（DB、署名された値、セッション）で行う。

### 3.3 入力ヘッダーの信頼境界を明示
クライアントから届く HTTP ヘッダーのうち、**何が信頼できる値で何が信頼できないか**を一段ずつ書き出す。デフォルトは「信頼しない」。

### 3.4 atomic な判定
「上限チェック → 利用記録」を分離してはならない。Race condition で即座に破られる。トランザクション、`SELECT ... FOR UPDATE`、`INSERT IGNORE`、原子カウンタ等で 1 動作にまとめる。

### 3.5 origin の隠蔽が前提
Cloudflare 等の CDN/WAF を抑止層に組み込む場合、**origin に CDN 以外から到達できないこと**が前提条件。これが破れている時点で CDN 由来のヘッダー判定はすべて無効化される。

### 3.6 識別キーは abuse 耐性で選ぶ
複数のシグナルを混ぜると公平性は上がるが、abuse 耐性は下がる場合がある。
- **Hard key**（authoritative）: 攻撃者がコストをかけないと変えられないシグナル（IP, 認証 ID）
- **Soft key**（補助）: 偽装容易だが NAT/共用環境を救うために使う（fingerprint, Accept-Language, UA）

Hard key を主、soft key を補助に倒す。逆にしてはならない。

### 3.7 残存リスクの受容範囲を文書化する
「この攻撃は止まらない」を曖昧にしない。コスト感とともに明記し、ビジネス側で受容判断する。

## 4. 信頼モデル（Trust Model）

### 4.1 信頼してよい値
| 値 | 条件 |
|---|---|
| `CF-Connecting-IP` ヘッダー | **origin が Cloudflare からしか到達できない** ことが保証されている場合のみ |
| 自社サーバーが署名した cookie/token | HMAC または JWT で検証できる場合のみ |
| 自社 DB の状態 | 常に信頼できる |
| Cloudflare Turnstile の siteverify レスポンス | レスポンス全体（success + hostname + action）を検証した場合のみ |

### 4.2 信頼してはならない値
| 値 | 理由 |
|---|---|
| `X-Forwarded-For` `X-Real-IP` | 誰でも自由に付けられる。CDN を経由していてもクライアントの値は混入する |
| `User-Agent` `Accept-Language` `Sec-CH-UA-*` | 偽装自由。偽装コストは文字列を変えるだけ |
| 値が固定文字列の cookie（例: `verified=1`） | 値が固定なので curl から自由に再現できる |
| 自社で生成した cookie で署名/期限がないもの | 上記と同じ |
| クライアント側で計算した fingerprint（FingerprintJS 無料版など） | 精度 40-60%。同一環境で衝突、ブラウザ更新で変化。**識別ではなく抑止の補助** にしか使わない |
| `localStorage` `sessionStorage` `IndexedDB` の値 | クライアントが自由に消去・編集できる |

## 5. 各層で達成すべきこと

| 層 | 役割 | 必須要件 |
|---|---|---|
| **Layer 0: Origin 隠蔽** | CDN バイパス遮断 | Cloudflare Tunnel または AOP + L3/L4 firewall allowlist。CDN レンジ以外からの 80/443 を drop |
| **Layer 1: Edge** | 既知の悪性トラフィックを早期遮断 | Cloudflare WAF（Tor、known VPN/データセンター ASN）+ Edge Rate Limit（短時間 burst） |
| **Layer 2: 人間判定** | Bot を篩い分ける | Cloudflare Turnstile（managed mode 推奨）、cookie は **必ず署名済 token** にする |
| **Layer 3: 識別キー** | 抑止カウンタの主キー | hard key = `CF-Connecting-IP`（IPv6 は `/64` 正規化）。soft key は補助のみ |
| **Layer 4: クオータ判定** | 1 リクエスト単位の最終判定 | atomic（Tx + 行ロック または `INSERT IGNORE` + UNIQUE 制約）、JST 0時境界で日次リセット（DB セッション TZ を固定） |
| **Layer 5: 監視・遮断** | 異常パターン検知 | 1 hash key からの異常 burst、特定 ASN からの集中ヒットを slack/メール通知。手動 ban 経路を持つ |

## 6. 設定原則（運用ルール）

### 6.1 環境変数
- 抑止用ソルト・署名鍵は **本番 `.env.local` のみ**に置き、レポジトリに入れない（`.gitignore` の `.env*` で保護）
- 鍵長は **32 bytes 以上の cryptographically secure random**（`openssl rand -hex 32`）
- 定期 rotation は不要。**漏えい疑い・privacy policy 上の理由がある場合のみ**回す
- 鍵未設定時は **production では起動失敗または該当 API を 503 返却**。フォールバックで「鍵なし=何でも通す」にしない

### 6.2 IPv6 の扱い
- IPv6 アドレスは `/128` 単位ではなく **`/64` 単位**で抑止キーに正規化する（多くの ISP/モバイルキャリアが /64 をユーザー単位で割り当てるため）

### 6.3 DB タイムゾーン
- 日次境界判定がある場合、DB セッションの timezone を **明示的に固定**（接続時 `SET time_zone = '+09:00'` 等）
- もしくは `consumed_at >= DATE_SUB(NOW(), INTERVAL 24 HOUR)` のような **境界が滑らかな** 条件に書き換える

### 6.4 ログとメトリクス
- 抑止が発火した（429/403）回数を時系列で観測する
- 「すり抜けた攻撃」を検出するため、1 hash key の連続成功数や 1 ASN の集中度をモニタリングする

### 6.5 残存リスクのコスト管理
- 全匿名 API について **「1 日に許容する LLM コスト総額」のキャップ**を別レイヤで設ける（個別 IP/Hash の制限とは別）
- マクロキャップに達したら全匿名 API を一時停止する経路を運用に組み込む

---

## 7. 受容する残存リスク

下記攻撃は本方針では止まらないことを明記する。受容するか、追加層（メール認証必須化等）で対応するかは事業判断。

| 攻撃 | コスト感 | 受容判断 |
|---|---|---|
| 住宅プロキシ網 + 実ブラウザ automation + Turnstile solver の組合せ | residential proxy ~$4–6/GB、Turnstile solver ~$1.45/1000 | LLM コスト次第ではペイする可能性あり。**マクロキャップで保険をかける** |
| Cloudflare 内部経路の特殊ケース（同一ゾーン Workers 経由など） | 設定不備時のみ | 設定レビューで個別検証 |
| 完全な認証突破・データ改ざん | スコープ外 | スコープ外 |

---

## 8. チェックリスト

新規・既存の匿名 API を本ポリシー準拠で運用するための最小チェック。

- [ ] origin が CDN/WAF 以外から HTTP/HTTPS で到達できないことを確認した
- [ ] `X-Forwarded-For` `X-Real-IP` 等のクライアント任意ヘッダーを抑止判定に使っていない
- [ ] hard key（IP）と soft key（fingerprint/UA 等）を分離している
- [ ] IPv6 を /64 で正規化している
- [ ] クオータ判定が atomic（race-free）である
- [ ] 日次境界が DB タイムゾーンに依存していない
- [ ] 抑止用 cookie/token が **署名されている**、または **使っていない**
- [ ] Turnstile siteverify では `success` + `hostname` + `action` を検証している
- [ ] ソルト・鍵が本番環境変数にあり、レポジトリにない
- [ ] 抑止発火数のメトリクスが監視できる
- [ ] LLM コスト総額のマクロキャップが別レイヤで設定されている
- [ ] 受容している残存リスクが文書化されている
